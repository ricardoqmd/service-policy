# 015 - Do not version the generated OpenAPI specification

- **Date:** 2026-07-06
- **Status:** Accepted
- **Deciders:** Ricardo Quintero
- **Refines:** —

## Context

The repository tracked `docs/openapi.json` and `docs/openapi.yaml`. These files are
generated by smallrye-openapi from the code annotations; they are build output, not
source. Because they were committed, they drifted from the code and — critically —
carried whatever `oidc.auth-server-url` the app was configured with when they were
generated. A local smoke run with OIDC pointed at a real IdP regenerated the specs and
baked an environment-specific auth-server URL into `openIdConnectUrl`, which reached the
working tree and was one `git add -A` away from a public commit. Secret scanners
(gitleaks) do not flag hostnames, so this class of leak is invisible to the current CI
gate.

## Decision

Stop versioning the generated OpenAPI specification. Remove `docs/openapi.json` and
`docs/openapi.yaml` from git tracking, add them to `.gitignore`, and rely on runtime
generation: the live contract is served at `/q/openapi` and written to the build's
`target/` output. The code annotations are the single source of truth for the contract.

## Reasons

- Generated artifacts are not source; committing them creates two sources of truth that
  drift.
- The spec captured environment-specific configuration (the IdP URL). Un-versioning
  removes the leak surface entirely: what is gitignored cannot be committed.
- The contract always matches the code, because it is derived from it on demand.

## Alternatives considered

- **Keep versioning, but regenerate deterministically from a docs profile with generic
  values.** Rejected: it still leaves a committable leak surface and relies on human
  discipline to always regenerate from the right profile — the discipline that just
  failed.
- **Keep versioning as-is.** Rejected: the leak already happened once.

## Consequences

- (+) No environment-specific value can enter the repository through the spec.
- (+) The published contract cannot drift from the implementation.
- (−) A reader browsing the repository cannot see the contract without running the app
  or CI. Mitigated by the recommendation below.

## Criteria to revisit

Revisit if external integrators or contract testing require a stable, reviewable spec
committed in-repo. In that case adopt the deterministic-publish path below rather than
re-committing raw generated files.

## Better option (impact if taken)

Publish the spec as a **CI artifact** (or GitHub Pages / release asset), generated in the
pipeline with a fixed docs profile whose `auth-server-url` is a generic placeholder.
Impact: restores contract visibility for a public/portfolio repository **without** the
leak surface or drift, at the cost of a small CI job. Recommended as a fast-follow — not
required to land this decision. Conservative move (un-version) ships now; the stronger
end-state (un-version + CI-published deterministic spec) is a small follow-up.
